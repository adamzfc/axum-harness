# Phase 14.1 Research — Deferred E2E stabilization and runtime cost

**Date:** 2026-04-07  
**Scope:** `.planning/phases/14-d-dev-rust-templates-tauri-sveltekit-axum-moon-template-docs/deferred-items.md` + `14.1-CONTEXT.md` locked decisions D-01..D-07  
**Outcome:** Ready for planning

## Summary

Phase 14.1 should close three concrete deferred gaps from 14-05 without changing product behavior:

1. Web Playwright tenant setup fails with `401` due to auth contract mismatch in fixture setup.
2. Web E2E intermittently fails with `ECONNREFUSED` because API readiness is not deterministic.
3. Full E2E cycle is slow because runtime startup/compile paths are repeatedly re-triggered.

Recommended implementation:

- Align web tenant fixture requests with backend auth contract (`Authorization: Bearer <token>`), reusing the tenant identities already used by desktop suite.
- Add deterministic runtime readiness orchestration (health check + controlled spawn/reuse) before web E2E runs.
- Introduce explicit preflight and runtime reuse path in repository gate commands to reduce repeated cold-start work.

## Codebase Findings

- Failing web fixture path:
  - `apps/client/web/app/tests/fixtures/tenant.ts` calls `POST http://127.0.0.1:3001/api/tenant/init` without auth headers.
  - Backend route (`servers/api/src/routes/tenant.rs`) documents `401` for missing/invalid JWT.
- Existing desktop suite already solved this class of issue:
  - `e2e-desktop-playwright/tests/specs/tenant-isolation.spec.ts` mints deterministic JWT-like test tokens and sends `Authorization` headers.
  - Same file includes API readiness polling + runtime_server spawn/reuse fallback.
- Full gate entry point:
  - `moon.yml` `test-e2e-full` currently runs web then desktop lanes sequentially, but web lane has no deterministic API preflight.

## Library/Tool Notes

### Playwright (v1.58.2)

Reference confirms these stable mechanisms for deterministic startup:

- `webServer` with `reuseExistingServer: !process.env.CI` for UI server lifecycle.
- `globalSetup` for custom dependency readiness logic (API checks, pre-test orchestration).
- Multi-server style orchestration can be done via explicit checks in setup scripts when non-HTTP process lifecycle is needed.

## Risk Analysis

### Risk 1 — Auth contract drift (web fixture vs backend)

- **Symptom:** `tenant init failed: status=401`.
- **Mitigation:** Make fixture send deterministic bearer token in exactly the same format used by desktop tenant-isolation helpers (D-01, D-06).

### Risk 2 — Non-deterministic runtime availability

- **Symptom:** `ECONNREFUSED 127.0.0.1:3001` at test bootstrap.
- **Mitigation:** Shared `ensure runtime ready` helper with bounded wait + controlled spawn + cleanup ownership guard (D-02, D-04).

### Risk 3 — Slow full-gate iteration due repeated runtime startup work

- **Symptom:** repeated long waits (cold compile/start) before tests can execute.
- **Mitigation:** preflight + reuse strategy in repo-level command path; prefer existing binary when available; avoid restarting owned process per spec (D-03).
