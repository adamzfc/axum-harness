# 13. 子领域选型研究指南

> **文档版本**: 1.0.0  
> **最后更新**: 2026-04-01

## 核心原则：避免重复造轮子

**在深入各个子领域时，优先使用成熟的库和框架，避免重复造轮子。**

当需要引入新的功能或技术时，应该：
1. 先 research 现有解决方案
2. 对比评估候选方案
3. 选择最适合的方案
4. 记录选型决策和原因

## 研究流程

### 1. 需求分析

在开始研究之前，明确需求：

- **功能需求**：需要实现什么功能
- **性能需求**：性能要求是什么
- **集成需求**：需要与现有系统如何集成
- **维护需求**：长期维护的考虑

### 2. 信息收集

从以下渠道收集信息：

| 渠道 | 说明 | 适用场景 |
|------|------|---------|
| **crates.io** | Rust 包仓库 | 查找 Rust 库 |
| **npm** | Node.js 包仓库 | 查找 JS/TS 库 |
| **GitHub** | 代码托管平台 | 查看项目活跃度、issue、PR |
| **Reddit r/rust** | Rust 社区 | 获取社区反馈和讨论 |
| **Hacker News** | 技术新闻 | 了解技术趋势和评价 |
| **技术博客** | 深度分析 | 获取详细的对比分析 |
| **官方文档** | 权威信息 | 了解 API 和最佳实践 |

### 3. 评估标准

评估候选方案时，考虑以下标准：

| 标准 | 权重 | 说明 |
|------|------|------|
| **功能完整性** | 高 | 是否满足所有功能需求 |
| **性能表现** | 高 | 性能是否满足需求 |
| **Rust 生态** | 高 | 是否是 Rust 生态的一部分 |
| **Agent 友好** | 高 | 文档和 API 是否适合 agent 学习 |
| **社区活跃度** | 中 | 是否有活跃的社区和维护 |
| **API 设计** | 中 | API 设计是否良好，易于使用 |
| **文档质量** | 中 | 文档是否完整、清晰 |
| **依赖数量** | 中 | 依赖是否过多，是否会增加复杂度 |
| **许可证** | 低 | 许可证是否兼容 |

### 4. 决策记录

选型决策应该记录在文档中，包括：

- **需求描述**：需要实现什么功能
- **候选方案**：考虑了哪些方案
- **选型结果**：选择了哪个方案
- **选择原因**：为什么选择这个方案
- **备选方案**：有哪些备选方案
- **风险评估**：有哪些潜在风险

## 避免重复造轮子的检查清单

在开始实现新功能之前，检查：

- [ ] 是否有现成的 Rust crate 可以使用
- [ ] 是否有现成的 JS/TS 包可以使用
- [ ] 是否有开源项目可以参考
- [ ] 是否有相关的技术博客或文档
- [ ] 社区是否有相关的讨论和解决方案

**如果以上都存在成熟的解决方案，优先使用它们，而不是自己实现。**

## 何时应该自己实现

在以下情况下，可以考虑自己实现：

1. **没有现成的解决方案**：确实没有满足需求的库
2. **现有方案过于复杂**：现有方案引入了过多不必要的依赖
3. **性能要求极高**：现有方案无法满足性能需求
4. **需要深度定制**：现有方案无法满足特定的定制需求
5. **学习目的**：为了学习和理解某个技术

**即使自己实现，也应该参考现有方案的设计和实现。**
