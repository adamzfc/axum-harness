## 这份蓝图解决什么问题

这不是一个"再加几个 crate/包"的建议集，而是一套用于指导长期迭代的工程母体蓝图。  
目标是把一个仓库建设成：

- 业务无关
- 跨宿主（desktop / web / extension / miniapp）
- 跨协议（HTTP / websocket / atproto / farcaster / nostr）
- 跨链（EVM / Base / Solana / TON）
- **Agent-Friendly 开发基建**：所有代码和开发由 agent 主导，基建要让 agent 开发更友好
- 对未来 3–5 年的技术变化保持高选项价值

## 技术选型哲学

### 核心理念：Rust 优先，前沿导向，Agent-Friendly

**我们选择 Rust 不是因为保守，而是因为它能创造出更好的产品。**

本蓝图的技术选型遵循以下原则：

#### 1. Rust 生态优先

- **为什么 Rust**：Rust 提供了无与伦比的性能、安全性和可靠性。用 Rust 构建的产品在性能、内存安全和并发处理上优于 Python、TypeScript、Java、Go 等语言。
- **Rust 后端**：从 Web 框架（Axum）到桌面应用（Tauri），Rust 生态已经成熟。
- **Rust 用于 AI/Agent**：Rig、candle 等 Rust AI 框架可以用于产品内的 AI 功能。

#### 2. 前沿技术拥抱

- **不选择"大厂保守实践"**：我们不选择因为"大厂在用"而选择的技术，而是选择因为"它是更好的解决方案"而选择的技术。
- **前沿但不实验性**：选择已经经过验证但仍在快速发展的前沿技术。
- **2026 年视角**：所有技术选型基于 2026 年 4 月的最新发展，而不是 2024 年的"最佳实践"。

#### 3. 解决长期痛点

- **选择融合旧技术优秀思想的新技术**：例如，Svelte 5 的 Runes 模式重新思考了前端响应式，Tauri v2 重新定义了桌面应用的安全性。
- **选择有长期发展能力的技术**：评估技术的社区活跃度、维护团队、发展路线图。

#### 4. Agent-Friendly 开发

**"Agent-Friendly" 不是指引入 AI Agent 框架，而是指基建要让 agent 开发更友好。**

我们的开发模式是：**代码和开发由 agent 主导完成**。因此基建需要：

- **文档清晰**：技术选择的文档要清晰、完整，agent 能快速学习
- **API 设计良好**：API 要有良好的类型系统，便于 agent 理解和使用
- **模式一致**：代码模式要一致，agent 能快速复制和应用
- **反馈及时**：编译错误、测试失败要快速反馈，agent 能快速迭代
- **避免重复造轮子**：优先使用成熟的库和框架，agent 不需要从零开始

### 技术选型可替代性

**所有技术选型都是独立、解耦、可替代的。**

技术选型不应该成为系统的耦合点。每个技术选择都应该：

1. **独立决策**：每个技术选型可以独立评估和替换
2. **接口隔离**：通过清晰的接口/契约隔离技术实现细节
3. **渐进演进**：技术栈可以渐进式升级，不需要大规模重构
4. **选项价值**：保持对未来更好技术的开放性

## 最小实现范围（V1）

### 功能清单

V1 最小实现包含以下功能：

1. **Google Auth**：用户可以通过 Google 账号登录
2. **Counter**：简单的计数器功能，验证前后端通信
3. **Admin Web**：管理后台界面
4. **Agent 对话**：用户可以通过导入 OpenAI 兼容的 API key，与产品内的 agent 进行对话

### Agent 对话功能

用户可以通过导入 OpenAI 兼容的 API key 来实现：

- **与产品内的 agent 进行对话**
- **Agent 可以操作产品功能**
- **Agent 可以操作用户授权的 workspace**

### Agent 技术选型

产品内的 AI 功能使用 Rust 生态：

| 功能 | 技术选择 | 说明 |
|------|---------|------|
| **LLM 调用** | OpenAI 兼容 API | 支持 OpenAI、Claude、本地模型等 |
| **Agent 框架** | 自研轻量级 | 基于 Rust，避免引入过重的框架 |
| **向量数据库** | libsql + 向量扩展 | 嵌入式支持，避免外部依赖 |
| **工具调用** | 自定义工具系统 | 与产品功能深度集成 |

## 适用范围

适合以下类型仓库：

- Tauri + SvelteKit + Rust 的跨平台应用
- 以 Bun 为 JS/TS runtime 和 package manager 的 monorepo
- 以 moon + proto + Justfile 作为顶层任务与工具链管理
- Agent-Friendly 开发基建
- 追求 Rust 生态优先、前沿技术导向的开发团队

## 非目标

这份蓝图 **不** 试图解决：

- 某个垂直业务域的模型设计
- 某个具体产品功能的 UX 细节
- 单一协议生态下的深度策略
- 立即穷尽所有未来技术

它只处理"几乎所有顶级 Rust / Tauri / SvelteKit / TS 项目都能复用"的工程问题。

## 文档版本管理

### 版本号规则

- **主版本号**：架构级变更，不兼容的修改
- **次版本号**：新增功能或重大改进，向后兼容
- **修订号**：小的修正和改进，向后兼容

### 更新流程

1. 识别需要更新的部分
2. 评估影响范围
3. 更新文档版本号
4. 更新"最后更新"日期
5. 记录变更历史

### 文档与代码的关系

**文档是规范，代码是实现。**

- 文档定义目标状态和规范
- 代码应该向文档靠拢
- 不应该反向修改文档以适应当前代码状态
- 当代码与文档不一致时，应该优先更新代码

## 推荐执行方式

建议你把这套文档当成仓库演化的"制度层"，而不是一次性的讨论结果。

执行顺序建议：

- 第一步：确认目录与边界
- 第二步：确认任务图与工具链职责
- 第三步：做 contracts/typegen 闭环
- 第四步：做观测、测试、evals 基线
- 第五步：实现最小功能（Google Auth、Counter、Admin、Agent 对话）

## 技术选型演进机制

### 持续评估

**不是每季度评估，而是持续关注。**

- **信息来源**：Reddit r/rust、Hacker News、Rust Blog、技术博客
- **评估标准**：性能、生态、社区活跃度、长期发展能力
- **决策方式**：基于数据和实际测试，而不是"大厂在用"

### 替换流程

当决定替换某个技术时：

1. **评估阶段**：全面评估新旧技术的差异
2. **试点阶段**：在小范围内试用新技术
3. **迁移阶段**：制定详细的迁移计划
4. **验证阶段**：确保迁移后系统正常运行
5. **清理阶段**：移除旧技术相关的代码和配置

### 风险控制

- 技术替换应该渐进式进行
- 保持向后兼容性
- 准备回滚方案
- 充分测试验证

## 反对"大厂保守实践"

### 为什么反对

- **大厂实践不一定适合独立开发者**：大厂有大量人力和资源，独立开发者需要更高效的工具。
- **大厂实践往往滞后**：大厂技术栈更新慢，往往使用几年前的"最佳实践"。
- **大厂实践不一定最优**：大厂选择技术考虑的因素更多（招聘、培训、迁移成本），不一定选择最优解。

### 我们的选择标准

- **性能优先**：选择性能最好的技术，而不是"最稳定"的技术
- **开发效率**：选择能让 agent 高效开发的技术
- **前沿但不实验**：选择已经经过验证但仍在快速发展的技术
- **Rust 生态**：优先选择 Rust 生态中的技术
